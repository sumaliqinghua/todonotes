# 决策与变更记录（DECISIONS）

## [M0.13-R21-simplify] 时间能力最终收敛为“仅截止时间”，且不兼容旧开始时间数据

- **改了什么**：
  - 删除开始时间语义，只保留正文块级 `dueAt` 截止时间。
  - Sticky 右键只保留 `设置截止时间...` 二级菜单和 `自定义截止时间...`。
  - 不做旧 `startAt` 数据兼容或迁移。
- **为什么这么改**：
  - 用户明确选择“极简版：只保留截止时间”。
  - 在当前产品里，手动待处理已经足够承担“以后再看”的语义；继续保留开始时间只会增加理解成本和状态复杂度。
- **已知限制**：
  - 历史实验数据里若存在 `startAt`，本轮起不会继续被识别。
  - 如果未来确实再次需要“晚点再开始”的能力，应作为新的独立需求重新设计，而不是把旧开始时间逻辑半恢复。
- **关联假设**：
  - `[2026-04-07/M0.13-R21-simplify] 时间能力收敛为“仅截止时间”`
  - `[2026-04-07/M0.13-R21-simplify] 不兼容旧 startAt 数据`

## [M0.13-R21] 将时间能力收束为 Sticky 的块级待处理工作台

- **改了什么**：
  - 在 `PRD.md` 中新增“带时间的待处理工作台”规划章节，明确时间属于正文块属性，而不是任务表字段或提醒表字段。
  - 在 `ASSUMPTIONS.md` 中固化本轮默认边界：保留手动待处理、快捷时间默认写截止时间、首轮只在 Sticky 落地、不支持 `taskLink` 时间。
  - 在 `plan.md` 中新增 R21 里程碑，把后续开发拆成“块级时间属性 → 右键入口 → 自定义时间窗口 → 时间筛选与切换 → 刷新与测试”五步。
- **为什么这么改**：
  - 当前项目里“待处理”“优先级”“提醒”已经各自承担了一部分“现在该做什么”的语义；如果直接再加一套时间能力，会继续放大入口分散问题。
  - Sticky 已经具备块级定位、待处理列表、左右连续跳转，是最接近“执行工作台”的现成承载体，把时间能力并入这里的改造成本和学习成本都最低。
- **已知限制**：
  - 这次决策只完成需求收束与计划拆分，尚未真正落地数据结构与界面。
  - Library 侧的 Today/1 小时内视图、提醒弹窗联动、任务级时间字段都被明确排除在本轮范围外。
- **关联假设**：
  - `[2026-04-07/M0.13-R21] 时间信息挂在正文块，不新增任务级时间字段`
  - `[2026-04-07/M0.13-R21] 保留手动待处理，并与时间块共用 Sticky 单一入口`
  - `[2026-04-07/M0.13-R21] 右键快捷时间默认写入截止时间，自定义窗口再选择时间类型`
  - `[2026-04-07/M0.13-R21] 首轮只在 Sticky 落地块时间工作台`

## [M0.13-R21-Impl] 时间条目按共享根任务汇总，并用真实 DOM 徽标渲染

- **改了什么**：
  - 新增主进程查询 `task:listTimedBlocksByRoot`，按 Sticky 共享根任务汇总整棵活跃子树中的带时间正文块。
  - 时间徽标不使用 CSS 伪元素，而是通过编辑器扩展直接同步到真实 DOM 节点。
- **为什么这么改**：
  - 只统计当前页无法形成真正的 Today / 1 小时内工作台，必须跨页面汇总。
  - 现有优先级已经占用了 `::before` 伪元素；如果时间也抢同一个位置，视觉会冲突，所以改成真实 DOM 徽标节点更稳。
- **已知限制**：
  - 当前时间条目跨页面汇总只在 Sticky 使用，Library 侧没有对应队列视图。
  - 真实 DOM 徽标是视图层能力，不会单独写入数据库；当前实际持久化的数据只有正文块属性里的 `dueAt`。
- **关联假设**：
  - `[2026-04-07/M0.13-R21] 时间队列按 Sticky 共享根整棵活跃子树汇总`
  - `[2026-04-07/M0.13-R21] 同一块同时存在手动待处理和时间时合并为一条`

## [M0.11] 同父任务便签共享状态并保留按路径打开语义

- **改了什么**：
  - 改造 `window:open` 的 sticky 打开逻辑：右键子任务打开便签时，自动回溯到顶层父任务作为共享根，同时将新窗口导航路径定位到被打开任务。
  - 改造 `windowManager`：sticky 创建时按共享根加载书签并自动补顶层父任务书签；移除“按书签包含关系猜测根任务”的兜底逻辑。
  - 新增事件 `window:sticky-shared-updated`，并在渲染层接入：同一共享根下修改书签/皮肤颜色/透明度后，多便签实时同步。
- **为什么这么改**：
  - 满足“同一个父级任务下便签共用同一状态对象，只有打开路径不同”的交互预期。
  - 避免以前按“当前打开任务”分裂便签状态，导致同一任务树下书签和皮肤不一致。
- **已知限制**：
  - 多父关系任务仍按祖先链首节点归属共享根，复杂图结构下可能与用户直觉不完全一致。
  - 当前同步范围聚焦 sticky 共享状态（书签/皮肤）；窗口几何与置顶仍按窗口实例保存。
- **关联假设**：
  - `[2026-02-08/M0.11] 便签共享范围按“祖先链首个父任务”归类`
